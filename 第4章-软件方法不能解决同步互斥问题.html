🧪 软件方法的困境 · 互斥失败演示

以「双标志 + 先检查后置标志」为例 —— 并发导致同时进入临界区

📌 进程 0 (P0)
1️⃣ while(flag[1]) ; // 检查对方是否想进入
2️⃣ flag[0] = true; // 置自己的标志
3️⃣ /* 临界区 */ count++ (共享变量)
4️⃣ flag[0] = false; // 退出
🔖 状态: ⏳ 未开始
🧾 flag[0] = false    flag[1] = false
📌 进程 1 (P1)
1️⃣ while(flag[0]) ; // 检查对方是否想进入
2️⃣ flag[1] = true; // 置自己的标志
3️⃣ /* 临界区 */ count++ (共享变量)
4️⃣ flag[1] = false; // 退出
🔖 状态: ⏳ 未开始
🧾 flag[0] = false    flag[1] = false
🔢 共享计数器 (critical resource): 0 ⚠️ 预期互斥: 每次只应一个进程进入
📋 操作记录:
  • ✨ 初始: flag[0]=false, flag[1]=false, counter=0
  • 💡 软件方法「先检查对方flag再置自己的flag」存在时间窗口
  • ⚠️ 若两个进程同时执行检查 → 都认为对方未进入 → 同时置标志 → 同时进入临界区 → 互斥失败
🔍 为什么软件方法注定失败？
• 这里演示的是「双标志先检查」法 (Dekker/Peterson之前的典型错误尝试)。
• 并发执行点：P0 执行完 while(flag[1]) 发现 false，尚未执行 flag[0]=true 时，
  P1 也执行 while(flag[0]) 发现 false → 两个进程都认为自己可以进入 → 然后各自设置 flag=true，
  最终同时进入临界区，破坏互斥。
• 手动执行或点击「自动演示冲突」会展示这种指令交错导致 counter 被两个进程同时修改(或丢失更新)，违背互斥。
🧪 教学演示：软件方法依赖特定执行顺序，无法避免并发调度导致的竞争条件 → 需要硬件/操作系统提供的原子操作。
